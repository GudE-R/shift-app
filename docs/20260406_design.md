### 2.2 テーブル定義詳細

#### OWNERS
Supabase Auth の `auth.users` と連携。`id` は `auth.users.id` と同一。

```sql
CREATE TABLE owners (
    id UUID PRIMARY KEY REFERENCES auth.users(id) ON DELETE CASCADE,
    email TEXT UNIQUE NOT NULL,
    display_name TEXT NOT NULL,
    created_at TIMESTAMPTZ DEFAULT now(),
    updated_at TIMESTAMPTZ DEFAULT now()
);
```

#### STORES
```sql
CREATE TABLE stores (
    id UUID PRIMARY KEY DEFAULT gen_random_uuid(),
    owner_id UUID NOT NULL REFERENCES owners(id) ON DELETE CASCADE,
    name TEXT NOT NULL,
    location TEXT,
    color TEXT NOT NULL DEFAULT '#3B82F6',
    business_hours JSONB NOT NULL DEFAULT '{}',
    sort_order INT NOT NULL DEFAULT 0,
    created_at TIMESTAMPTZ DEFAULT now(),
    updated_at TIMESTAMPTZ DEFAULT now()
);

-- business_hours の想定構造:
-- {
--   "0": { "open": "09:00", "close": "22:00" },  -- 日曜
--   "1": { "open": "07:00", "close": "23:00" },  -- 月曜
--   ...
--   "6": { "open": "09:00", "close": "22:00" }   -- 土曜
-- }
```

#### STAFF
```sql
CREATE TABLE staff (
    id UUID PRIMARY KEY DEFAULT gen_random_uuid(),
    owner_id UUID NOT NULL REFERENCES owners(id) ON DELETE CASCADE,
    display_name TEXT NOT NULL,
    status TEXT NOT NULL DEFAULT 'active'
        CHECK (status IN ('active', 'leave', 'retired')),
    night_shift_ok BOOLEAN NOT NULL DEFAULT true,
    max_consecutive_days INT,
    monthly_hours_target DECIMAL(5,1),
    monthly_hours_min DECIMAL(5,1),
    monthly_hours_max DECIMAL(5,1),
    memo TEXT,
    created_at TIMESTAMPTZ DEFAULT now(),
    updated_at TIMESTAMPTZ DEFAULT now(),
    CHECK (monthly_hours_min <= monthly_hours_target),
    CHECK (monthly_hours_target <= monthly_hours_max)
);
```

#### AVAILABILITY
```sql
CREATE TABLE availability (
    id UUID PRIMARY KEY DEFAULT gen_random_uuid(),
    staff_id UUID NOT NULL REFERENCES staff(id) ON DELETE CASCADE,
    day_of_week INT NOT NULL CHECK (day_of_week BETWEEN 0 AND 6),
    status TEXT NOT NULL DEFAULT 'available'
        CHECK (status IN ('available', 'maybe', 'unavailable')),
    available_from TIME,
    available_to TIME,
    created_at TIMESTAMPTZ DEFAULT now(),
    updated_at TIMESTAMPTZ DEFAULT now(),
    UNIQUE (staff_id, day_of_week)
);
```

#### SHIFT_ENTRIES
```sql
CREATE TABLE shift_entries (
    id UUID PRIMARY KEY DEFAULT gen_random_uuid(),
    shift_period_id UUID NOT NULL REFERENCES shift_periods(id) ON DELETE CASCADE,
    staff_id UUID NOT NULL REFERENCES staff(id) ON DELETE CASCADE,
    store_id UUID NOT NULL REFERENCES stores(id) ON DELETE CASCADE,
    position_id UUID REFERENCES positions(id) ON DELETE SET NULL,
    work_date DATE NOT NULL,
    start_time TIME NOT NULL,
    end_time TIME NOT NULL,
    is_ai_generated BOOLEAN NOT NULL DEFAULT false,
    created_at TIMESTAMPTZ DEFAULT now(),
    updated_at TIMESTAMPTZ DEFAULT now(),
    CHECK (start_time < end_time)
);

CREATE INDEX idx_shift_entries_period ON shift_entries(shift_period_id);
CREATE INDEX idx_shift_entries_staff_date ON shift_entries(staff_id, work_date);
CREATE INDEX idx_shift_entries_store_date ON shift_entries(store_id, work_date);
```

#### CONSTRAINT_SETTINGS
```sql
CREATE TABLE constraint_settings (
    id UUID PRIMARY KEY DEFAULT gen_random_uuid(),
    owner_id UUID NOT NULL REFERENCES owners(id) ON DELETE CASCADE,
    rule_key TEXT NOT NULL,
    enabled BOOLEAN NOT NULL DEFAULT true,
    params JSONB NOT NULL DEFAULT '{}',
    updated_at TIMESTAMPTZ DEFAULT now(),
```

### 2.3 RLS（Row Level Security）ポリシー

全テーブルに共通して、オーナー自身のデータのみアクセス可能にする。

```sql
-- 例: stores テーブル
ALTER TABLE stores ENABLE ROW LEVEL SECURITY;

CREATE POLICY "owners_stores_select" ON stores
    FOR SELECT USING (owner_id = auth.uid());

CREATE POLICY "owners_stores_insert" ON stores
    FOR INSERT WITH CHECK (owner_id = auth.uid());

CREATE POLICY "owners_stores_update" ON stores
    FOR UPDATE USING (owner_id = auth.uid());

CREATE POLICY "owners_stores_delete" ON stores
    FOR DELETE USING (owner_id = auth.uid());

-- staff, shift_periods 等も同様に owner_id ベースで制御
-- staff_stores, availability 等の子テーブルは
-- staff.owner_id 経由で EXISTS サブクエリにより制御
```

子テーブル（staff 配下）のRLS例:

```sql
CREATE POLICY "owners_availability_select" ON availability
    FOR SELECT USING (
        EXISTS (
            SELECT 1 FROM staff
            WHERE staff.id = availability.staff_id
            AND staff.owner_id = auth.uid()
        )
    );
```

```typescript
import { GoogleGenAI } from '@google/genai';

const ai = new GoogleGenAI({ apiKey: process.env.GEMINI_API_KEY! });

async function generateShift(prompt: string): Promise<ShiftEntry[]> {
  const response = await ai.models.generateContent({
    model: 'gemini-2.5-pro',
    contents: prompt,
    config: {
      responseMimeType: 'application/json',
      responseSchema: shiftResponseSchema,  // JSON Schema で型を強制

  });
```

### 5.2 バリデーション実行タイミング

| タイミング | 実行場所 | 目的 |
|-----------|----------|------|
| AI生成直後 | サーバーサイド | 生成結果の品質チェック + 警告生成 |
| D&D操作時 | クライアントサイド（Zustand） | リアルタイムフィードバック |
| シフト確定時 | サーバーサイド | 最終チェック（未解決 error があれば確定ブロック） |

## 6. セキュリティ設計

| 対策 | 実装 |
|------|------|
| 認証 | Supabase Auth（メール+パスワード、JWT） |
| 認可 | PostgreSQL RLS（全テーブルに `owner_id = auth.uid()` ポリシー） |
| 通信暗号化 | Vercel / Supabase ともに HTTPS 強制 |
| 入力バリデーション | Zod による API 入力スキーマ検証 |
| APIキー管理 | Vercel 環境変数（`GEMINI_API_KEY`, `SUPABASE_*`） |
| CSRF対策 | Next.js のデフォルト保護 + SameSite Cookie |
| SQLインジェクション | Supabase クライアント経由のパラメータ化クエリ |

## 7. デプロイメント・CI/CD

### 7.1 環境構成

| 環境 | URL | 用途 |
|------|-----|------|
| 開発 | `localhost:3000` | ローカル開発 |
| プレビュー | `*.vercel.app` | PR ごとの自動プレビュー |
| 本番 | `shiftcraft.vercel.app`（仮） | 本番環境 |

### 7.2 環境変数

```env
# Supabase
NEXT_PUBLIC_SUPABASE_URL=https://xxx.supabase.co
NEXT_PUBLIC_SUPABASE_ANON_KEY=xxx
SUPABASE_SERVICE_ROLE_KEY=xxx

# Gemini
GEMINI_API_KEY=xxx
```

### 7.3 CI/CD パイプライン

> **図**: [20260406_diagrams.md §11](20260406_diagrams.md#11-cicdパイプライン) を参照

- **自動デプロイ**: `main` ブランチ push → Vercel 本番デプロイ
- **プレビュー**: PR 作成 → Vercel プレビュー環境自動生成
- **DBマイグレーション**: `supabase db push` で手動適用（本番は慎重に）

## 8. パフォーマンス設計

| 項目 | 方針 |
|------|------|
| 初期表示 | RSC（React Server Components）でサーバーサイドレンダリング |
| シフトカレンダー | クライアントコンポーネント。Zustand で状態管理、操作ごとの差分更新 |
| AI生成 | API Route でストリーミングは不要（構造化JSON一括返却）。タイムアウト60秒 |
| D&D操作 | Optimistic Update。API応答を待たずUI即時反映 |
| データ取得 | 月単位でフェッチ。React Server Components の `fetch` キャッシュ活用 |
| PDF生成 | サーバーサイドでレンダリング。大規模データ時は Vercel Serverless の 10秒制限に注意 → 必要に応じて Edge Function |
