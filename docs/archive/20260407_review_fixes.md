## 経緯

本文書は、各レビューで検出された指摘事項と、それに対する修正内容をまとめたものである。

## 第1回レビュー（Claude）: 12件

初回ドラフトにおけるデータモデルの不整合と設計の欠落を中心に指摘。

| # | 重要度 | 指摘 | 修正内容 |
|---|--------|------|----------|
| 1 | 高 | MVPが重すぎる（Tauri + Expo + E2EE + AI + PDF を全部含む） | Phase 1〜3 に分割。Phase 1 はデスクトップ単体・同期なし |
| 2 | 高 | `出勤可能店舗` `対応可能職種` が UUID[] だが SQLite に配列型がない | `staff_stores` / `staff_positions` 中間テーブルを全ドキュメントに追加 |
| 3 | 中 | POSITIONS の CREATE TABLE 定義が設計書に欠落 | 設計書 §2.1 に追加 |
| 4 | 中 | 月間目標時間が要件書では3値、設計書では1カラム | `target_hours` / `min_hours` / `max_hours` の3カラムに統一 |
| 5 | 中 | 曜日別出勤可否の格納先テーブルがない | `staff_availability` テーブルを追加 |
| 6 | 中 | 連続勤務上限：要件書「個別設定」vs 設計書「5日固定」 | スタッフ個別の `max_consecutive_days` カラムに統一、デフォルト5日 |
| 7 | 中 | 絶対NG日（最優先制約）の格納先テーブルがない | `staff_ng_dates` テーブルを追加 |
| 8 | 中 | ER図に STAFF ↔ STORES の多対多リレーションがない | `STAFF_STORES` / `STAFF_POSITIONS` をER図に追加 |
| 9 | 高 | 同期コンフリクト解決戦略が未定義 | LWW + 手動修正優先ルールを設計書 §5.3 に追記、シーケンス図を追加 |
| 10 | 高 | 同期のマージ粒度（DB全体 or レコード単位）が不明 | レコード単位の変更追跡を設計書 §5.2 に明記 |
| 11 | 中 | `shift_entries` に `position_id` カラムがない | 設計書の CREATE TABLE に追加 |
| 12 | 中 | Geminiプロンプト設計がほぼ空（方針のみ） | レスポンスJSON形式・エラーハンドリング表を設計書 §3.2 に追記 |

## 第2回レビュー（Gemini）: 4件

第1回修正後のドキュメントに対し、同期整合性とAPI仕様の用語統一を指摘。

| # | 重要度 | 指摘 | 修正内容 |
|---|--------|------|----------|
| G1 | 中 | `updated_at` が `shift_entries` にしかない | 全テーブル（stores, staff, positions, 中間テーブル含む）に `updated_at` を追加 |
| G2 | 低 | ER図に `updated_at` が反映されていない | ER図の全エンティティに `updated_at` カラムを追加 |
| G3 | 中 | APIリクエストの `requirements` が `any` のまま | `stores[].requirements[].slots` として store_id / position_id / count を含む構造に具体化 |
| G4 | 低 | APIリクエストの `skills` が `staff_positions` と用語不一致 | `position_ids: string[]` に統一 |

## 第3回レビュー（Claude）: 6件

第2回修正後のドキュメントに対し、残存する細部の不整合を指摘。

| # | 重要度 | 指摘 | 修正内容 |
|---|--------|------|----------|
| D1 | 低 | 画面遷移図に「必要人員設定」画面がない | 画面遷移図に `Requirements[必要人員設定]` を追加 |
| D2 | 低 | `CONFLICT_LOGS` が `SHIFT_ENTRIES` のみに紐づくリレーションになっている | リレーション線を削除。CONFLICT_LOGS は `table_name` + `record_id` で任意テーブルを参照する汎用設計であることをコメントで明示 |
| D3 | 中 | APIリクエストに `min_hours` / `max_hours` が含まれていない | 設計書 §3.1 のリクエスト型に `min_hours` / `max_hours` を追加 |
| D4 | 中 | 必要人員設定の「曜日指定テンプレート」がスキーマと不整合 | Phase 1 では日付指定のみに限定（要件書から曖昧表現を削除）。テンプレート機能は Phase 3 に移動 |
| D5 | 低 | 設計思想テーブルに「スキルのパズル」が残存 | 「ポジション条件のパズル」に修正 |
| D6 | 中 | マスタデータ（stores/staff等）のコンフリクト解決基準が未記載 | マスタデータは一律 LWW であることを設計書 §5.3 に明記（シフトデータとルールを分離） |

## 修正後の状態

- **要件定義書 ↔ 設計書**: データモデル（テーブル・カラム）が完全に一致。
- **要件定義書 ↔ 図解集**: 画面一覧と画面遷移図が一致。ER図が全テーブルを網羅。
- **設計書 ↔ 図解集**: CREATE TABLE 定義と ER図のカラムが一致。
- **用語**: 「スキル」→「対応可能ポジション」/ `position_ids` に統一済み。
- **スコープ**: Phase 1（MVP）の範囲が明確化。テンプレート・統計は Phase 3 へ。
